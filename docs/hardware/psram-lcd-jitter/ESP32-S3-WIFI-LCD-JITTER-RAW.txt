Answering the 7 open research questions
1. Is CONFIG_SPIRAM_MODE_OCT the missing piece?
Yes, definitively. It's not one piece of the fix — it's the prerequisite for everything else. The Espressif FAQ data makes this concrete: at 30 MHz PCLK, you need Octal PSRAM at 80 MHz minimum to stay near the safe operating envelope, and Octal at 120 MHz to have headroom for WiFi contention. Quad at 80 MHz supports a maximum of ~11 MHz PCLK — below that, the system is technically in bus starvation every single frame.
The custom_sdkconfig + __wrap_log_printf blocker is real and well-understood: the pre-compiled Arduino WiFi .a blobs were built against a specific ESP-IDF configuration, and any change to PSRAM mode or XIP settings alters the linker-level symbol wrapping, breaking compatibility. There is no clean workaround within the pure Arduino framework path.
2. Does SPIRAM_TRY_ALLOCATE_WIFI_LWIP=n eliminate WiFi DMA on the PSRAM bus?
This is your highest-leverage fix that's potentially achievable today without a framework change. If you can set it, WiFi packet buffers move entirely to internal SRAM — no WiFi DMA ever touches the PSRAM bus again. The cost is ~50–100 KB of internal SRAM for WiFi buffers, which the ESP32-S3's 320 KB SRAM can absorb. The critical unknown is whether this particular sdkconfig option triggers the __wrap_log_printf error; unlike PSRAM mode, changing WiFi buffer placement shouldn't alter symbol wrapping. Worth testing in isolation before anything else.
3. What do commercial products and Espressif's own reference boards do?
The Espressif ESP-BSP for the ESP32-S3-LCD-EV-Board explicitly identifies bounce buffer mode as the recommended fix specifically for WiFi + LCD coexistence: "Enabling bounce buffer mode can lead to a higher PCLK frequency at the expense of increased CPU consumption. This mode is particularly useful when dealing with screen drift, especially in scenarios involving Wi-Fi usage or writing to Flash memory. This feature should be used in conjunction with ESP32S3_DATA_CACHE_LINE_64B configuration." GitHub
The coupling of bounce buffer mode with DATA_CACHE_LINE_64B is non-negotiable — Espressif's FAQ states that without 64B cache lines, bounce buffer mode itself "may cause the screen to drift." Your current config has DATA_CACHE_LINE_32B, which means the bounce buffer mode you've already implemented is operating incorrectly per Espressif's own spec. This is a high-value, low-risk fix.
The ESP32-S3-LCD-EV-Board example notes that "the PSRAM 120M DDR feature is intended to achieve the best performance of RGB LCD." GitHub Espressif's reference product targets 120 MHz PSRAM for LCD applications.
4. Is framework = espidf, arduino (hybrid mode) a viable path?
A confirmed working community post for an ESP32-S3-DevKitC-1-N16R8 board shows that with framework = arduino, espidf, setting CONFIG_SPIRAM=y and CONFIG_SPIRAM_MODE_OCT=y successfully activates Octal PSRAM, confirming "Bus Mode: OPI." PlatformIO Community This is the lowest-cost path to OCT mode that preserves Arduino APIs.
The migration cost is moderate: you keep your Arduino WiFi.h, Serial, etc., but ESP-IDF is compiled from source alongside it, so the linker symbol conflict disappears. The cert file issue (.crt.S files) still applies and needs a pre-build script, but it's a one-time fix, not an ongoing problem. The main risk is that framework = arduino, espidf has a somewhat narrower community than pure Arduino — you're working closer to the metal — but it has been successfully used for exactly this class of problem.
5. Can the bounce buffer ISR be made faster?
Yes, through a combination of:

CONFIG_LCD_RGB_ISR_IRAM_SAFE=y — keeps the ISR in IRAM so flash cache misses can't stall it mid-copy. The Espressif FAQ explicitly lists this as a fix for screen drift when flash operations occur. Esptool Documentation
CONFIG_ESP32S3_DATA_CACHE_LINE_64B=y — larger cache lines mean the CPU can fetch more PSRAM data per cache transaction, reducing the number of round-trips in the bounce buffer copy loop.
CONFIG_COMPILER_OPTIMIZATION_PERF=y — -O2 instead of -Os generates faster memcpy code. This specifically benefits the inner loop of the bounce buffer ISR.
XIP from PSRAM — eliminates flash cache misses entirely during ISR execution.

6. Is there an ESP-IDF GDMA priority mechanism for LCD DMA?
This is a hardware limitation with no clean software workaround: "The only way to solve this would be some sort of hardware priority schemes which allow GDMA to complete its job without being interrupted." No public API exists in ESP-IDF 5.x to elevate LCD GDMA priority over WiFi GDMA. ESP32 Forum HAL-level register writes are theoretically possible but fragile and unsupported. The rx_weight/tx_weight registers exist but aren't exposed. Don't pursue this path.
7. Does ESP-IDF 5.5+ have new LCD+WiFi coexistence features?
The most useful runtime API is esp_lcd_rgb_panel_set_pclk(), which dynamically adjusts the pixel clock. Espressif's FAQ explicitly recommends using this to throttle PCLK during WiFi-heavy operations: reduce it to something like 6 MHz before the operation, delay ~20 ms for one frame to complete, then restore PCLK. This causes a brief blank flash but prevents permanent drift. Esptool Documentation You can hook this into FreeRTOS event groups or callbacks around your Redis heartbeat publishes — lower PCLK, publish, restore. The dma_burst_size field in esp_lcd_rgb_panel_config_t replaces the deprecated alignment fields and should be set to 64 for optimal alignment with 64B cache lines.

What the evidence points to
The ESP32-S3 RGB LCD driver has three relevant operating patterns: full frame buffer in PSRAM, double frame buffer in PSRAM, and bounce-buffer mode. In bounce-buffer mode, the LCD DMA reads internal SRAM while the CPU copies from PSRAM into the bounce buffer; Espressif notes this can flicker or fail when PSRAM bandwidth is stressed, especially if CPU and another core are both accessing PSRAM through cache.

That aligns with the issue document’s core hypothesis: WiFi traffic plus LCD refresh are colliding on the same external-memory path, so reducing UI work helps but does not fully solve the problem when WiFi remains active.

PSRAM bandwidth is saturated or near-saturated by the display path alone, leaving too little headroom for WiFi bursts.

WiFi/LwIP buffers may be landing in PSRAM, increasing contention further.

Flash-cache stalls can delay bounce-buffer refill work, making a visually minor delay turn into a missed buffer swap or horizontal shift.

The current build appears to be using settings that do not match the hardware’s OPI PSRAM capability, which can leave performance on the table.
